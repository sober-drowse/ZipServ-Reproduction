# 个人实践总结

## 1. 本次复现学到的技术

本次复现让我对 GPU kernel benchmark、CUDA 编译环境、LLM 推理中的权重压缩和实验工程管理有了更具体的认识。ZipServ 的核心思想不是单纯追求压缩率，而是把压缩格式设计成适合 GPU 并行解码和 Tensor Core 计算的形式，从而让无损压缩有机会转化为推理速度收益。

具体收获包括：

- 理解了 BF16 数据格式中 sign、exponent、mantissa 的作用，以及为什么 LLM 权重的 exponent 分布会影响无损压缩设计。
- 学会了在服务器上安装 Miniconda、创建独立 conda 环境，避免污染其他同学的环境。
- 熟悉了 `nvidia-smi`、`nvcc --version`、`make`、`LD_LIBRARY_PATH` 等 GPU 实验常用命令。
- 学会了把服务器实验代码、日志、结果和图表同步到 GitHub 仓库，并在本地保留备份。
- 了解了 CUDA host/device 函数限制，尤其是 BF16 intrinsic 在不同 CUDA 版本中的兼容性问题。
- 学会了用脚本批量运行实验、解析日志、生成 CSV 和复刻图表。

## 2. 遇到的共性问题

复现过程中最明显的问题是论文环境与实际服务器环境不完全一致。官方代码默认 `/usr/local/cuda`，但服务器的 `nvcc` 位于 `/usr/bin/nvcc`；官方代码在当前 CUDA 12.0 中还存在 BF16 host/device 编译兼容问题。这说明论文代码即使开源，也不一定能在不同服务器上直接运行。

另一个问题是实验资源限制。论文中的主实验覆盖多种 GPU、多个模型、多个 baseline 和端到端推理框架，而本次只有当前 L20 服务器可用。因此复现时需要合理划分优先级，先完成官方 kernel benchmark 的最小闭环，再逐步扩展到 N sweep 和 shape sweep。

## 3. 对论文实验写作的启发

这次复现让我意识到，论文中的实验不仅要展示最终性能，还应该尽量提供清楚的运行命令、硬件环境、编译参数、baseline 获取方式和数据处理脚本。否则读者可能能理解方法，但很难完整复现结果。

对于自己的后续论文阅读和实验写作，我会特别注意：

- 明确区分“方法验证实验”和“主结论实验”。
- 保留完整日志，而不是只截图最终结果。
- 把环境、依赖、编译参数写成可执行脚本。
- 对复现失败或结果不一致的地方做客观解释，而不是只报告成功结果。
- 在图表中说明硬件、输入 shape、batch/token 参数，避免指标脱离上下文。

## 4. 本次复现的总体感受

本次复现最终跑通了 ZipServ 官方 kernel benchmark 的核心流程，并形成了代码、日志、结果表、图表和报告材料。虽然没有完整复现论文所有端到端实验，但已经完成了一个从论文阅读、服务器环境搭建、代码编译修复、实验运行、数据整理到结果汇报的闭环。

对我来说，最大的收获是理解了“复现”不只是把代码跑起来，而是要把实验逻辑、环境差异、结果可信度和复现限制都讲清楚。
